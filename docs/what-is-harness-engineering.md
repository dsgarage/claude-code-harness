# Harness Engineering とは？

## 一言でいうと

**AI エージェントの出力品質を「お願い」ではなく「仕組み」で保証するエンジニアリング手法。**

## 背景：なぜ必要か？

Claude Code のような AI エージェントは、コードの読み書き・コマンド実行・ファイル操作ができる強力なツール。
しかし、プロンプトで「テストを書いて」「危険なコマンドは避けて」と頼むだけでは：

- **確率的** — 99% 守っても 1% 守らない
- **セッションごとにリセット** — 前回の約束を覚えていない
- **コンテキスト依存** — 長いやり取りの中で指示を忘れる

Harness Engineering は、この「確率的な遵守」を **決定論的な強制** に変える。

## 核心的な洞察

> **同じモデルでも、ハーネスを変えるだけでスコアが 22 ポイント変動する。
> モデルを交換しても 1 ポイントしか変わらない。**
> — Morph 分析（SWE-bench）

つまり、どのモデルを使うかより、**どう制御するか** のほうが圧倒的に重要。

## 7つの柱

### 1. リポジトリ衛生 — 腐敗を前提に設計する

エージェントはリポジトリ内のテキストを等しく信頼する。
古いドキュメントがあると、それに従って間違ったコードを書く。

**対策：**
- 説明文書より **テスト** で仕様を表現する（テストは実行すれば腐敗がわかる）
- **ADR**（Architecture Decision Records）でステータスを明示する
- 古いドキュメントは削除するか、テストに置き換える

### 2. 決定論的ツールでガードレール — Hooks システム

```
「テストを書いてね」（プロンプト）→ 確率的
 テスト通過までエージェントを止める（Hook）→ 決定論的
```

Claude Code の Hooks は、エージェントのライフサイクルに介入する 4 つのポイント：

| Hook | タイミング | 用途 |
|------|-----------|------|
| **PreToolUse** | ツール実行 **前** | 危険操作ブロック、設定ファイル保護 |
| **PostToolUse** | ツール実行 **後** | リンター/フォーマッター自動実行 |
| **Stop** | エージェント完了宣言時 | テスト自動実行、未通過なら続行 |
| **PreCompact** | コンテキスト圧縮前 | 重要情報の保護 |

→ 詳細は [hooks-guide.md](hooks-guide.md) を参照

### 3. CLAUDE.md はポインタとして設計する

**書くべきもの：**
- テスト/ビルドのコマンド（`npm test`, `xcodegen generate`）
- 禁止事項リスト（リンタールールへの参照付き）
- ADR の保存場所

**書くべきでないもの：**
- システムの説明（コードとテストが真実）
- コーディングスタイルガイド（リンターに委ねる）

**目安：50行以下。** 200行を超えると性能が劣化し始める。

### 4. 計画と実行を分離する

エージェントに一度に全てをやらせない。

1. まず **計画** を立てさせる（Plan Mode）
2. 人間が **レビュー・承認**
3. 承認後に **実行**

### 5. E2E テストでエージェントに「目」を与える

コードを書くだけでなく、実際に動かして結果を見せる。

- **Web**: Playwright CLI / agent-browser
- **モバイル**: XcodeBuildMCP / mobile-mcp / Detox
- **CLI**: bats-core / pexpect
- **API**: Hurl / Pact

### 6. セッション間の状態管理

各セッションは前回の記憶がない。対策：

- Git コミットメッセージを「次のセッションへの引き継ぎ」として書く
- 進捗記録は Markdown より **JSON**（モデルが不適切に編集しにくい）
- 起動ルーチンを標準化（git log 読み → タスク選択 → サーバー起動）

### 7. プラットフォーム固有の最適化

Claude Code の強みは **Hooks による決定論的品質ゲート**。
ファイル編集のたびにリンター実行、完了のたびにテスト実行を強制できる。

## フィードバック速度の階層

```
PostToolUse Hook (ミリ秒)     ← 最速。ここで捕まえるのが理想
  → プリコミットフック (秒)
    → CI/CD パイプライン (分)
      → 人間レビュー (時間)
```

**できるだけ速いレイヤーにチェックを移動** するのが Harness Engineering の原則。

## アンチパターン（やってはいけないこと）

| やりがち | 正しいアプローチ |
|---------|----------------|
| 「テスト書いてね」とプロンプト | Stop Hook でテスト通過を強制 |
| CLAUDE.md に 1000 行の説明 | 50行のポインタ + リンター |
| リポジトリに設計文書を蓄積 | テスト + ADR で表現 |
| エージェント専用インフラ構築 | 人間にも良い開発インフラを構築 |

## このリポジトリとの関係

このリポジトリ `claude-code-harness` は、上記の原則を実践するための設定集：

- `settings.json` → Hooks 定義（柱 2）
- `CLAUDE.md` → ポインタ設計（柱 3）
- `hooks/safe_command.py` → PreToolUse 安全ゲート（柱 2）
- `skills/` → 定型タスクの再利用（柱 4, 5）
